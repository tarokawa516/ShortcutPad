# 引き継ぎ情報

## 目的

ShortcutPadの目的は、頻度の高いWindowsショートカットを、マウスやタッチ操作から最小の手数で実行できるようにすることです。

フル機能のスクリーンキーボードを目指すのではなく、日常的に繰り返す操作を少数の分かりやすいボタンへ絞ることを重視します。

## 出発点

2026年7月、SplashtopでスマートフォンからWindowsを操作する際、`Ctrl+A`、`Ctrl+C`、`Ctrl+Shift+V`などの入力が煩雑であることをきっかけに試作しました。C#とWinFormsを選んだ理由は、既存のVisual Studio環境を利用でき、Windows固有のウィンドウ制御とキー入力APIを小さな構成で扱えるためです。

その後、リモートアクセス時に限らず、通常のデスクトップ操作でも実用的であることが確認され、独立したプロジェクトとして継続することになりました。

## 現在の設計方針

1. ボタン数を増やしすぎず、頻度の高い操作を優先する。
2. タッチでも押しやすい大きさと間隔を保つ。
3. パネルの操作で、キー送信先アプリのフォーカスを奪わない。
4. 常時表示しても邪魔になりにくいよう、移動と折り畳みを可能にする。
5. キー送信の意味は対象アプリに委ね、アプリ固有の差を明示する。

## 実装済み

- WinFormsによるボーダーレス・常時手前表示パネル
- `WS_EX_NOACTIVATE`による非アクティブウィンドウ
- Windows `SendInput` APIによる修飾キーと通常キーの送信
- `Ctrl+A`から80ミリ秒後に`Ctrl+C`を送る複合操作
- 高DPI対応
- パネル移動、折り畳み、終了

## 確認済み

- .NET 10 SDKでビルド成功
- ビルド時の警告0、エラー0
- 試作版を一定期間、実作業で使用

## 既知の制約

- 管理者権限のアプリへは、通常権限で起動したShortcutPadから入力できない場合がある。
- `Ctrl+Shift+V`の挙動はアプリごとに異なる。
- ボタン、配置、ウィンドウ位置は現在コードに固定されている。
- 設定の保存、自動起動、通知領域、配布パッケージは未実装。
- 自動テストは未整備。OSへ実際のキー入力を送るため、結合テスト方法を検討する必要がある。

## 次の改善候補

優先順位は実際の使用感から決めます。候補は次のとおりです。

- ボタン内容と並び順の設定
- ウィンドウ位置、サイズ、折り畳み状態の保存
- 小型レイアウトや縦・横レイアウトの切り替え
- Windows起動時の自動起動
- 通知領域からの表示・非表示
- 単一インスタンス化
- 配布用の自己完結型ビルドまたはインストーラー
- アクセシビリティ名、配色、押下フィードバックの改善

## 変更時に守る点

- `WS_EX_NOACTIVATE`と`ShowWithoutActivation`を外すと、ボタンを押した時点で送信先がShortcutPad自身になる可能性がある。
- `SendInput`では修飾キーを押した順と逆順で必ず解放する。
- 複合操作の待ち時間はアプリによって必要量が異なるため、変更時は複数アプリで確認する。
